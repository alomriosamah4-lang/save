# سيناريو تجربة المستخدم لمشروع "خزنتي" — متطلبات، خريطة تنفيذ، وخطة تحقق

هذا المستند يربط سيناريو تجربة المستخدم المفصّل مع متطلبات التطبيق، التبعيات الفنية، ونقاط التحقق (Acceptance Criteria + Tests) لكل خطوة.

ملاحظات عامة:
- نفّذ كل خطوة ثم قم بالتحقق العميق (unit/integration/manual) قبل الانتقال للخطوة التالية.
- عند أي عملية تتعامل مع أسرار (مفاتيح، كلمات مرور) يجب ألا تُخزّن البيانات الحساسة كـ `byte[]` في Java heap؛ استخدم native/Keystore/session handles.

## 1️⃣ فتح التطبيق لأول مرة (Splash -> Start)

- واجهة: شاشة ترحيب مع شعار، نص تعريف وزر `بدء الاستخدام`.
- متطلبات فنية: شاشة بسيطة React Native (`Splash`, `Onboarding`).
- تحقق:
  - الوحدة: تواجد المكونات وفاعلية زر `بدء الاستخدام` ينتقل لشاشة إعداد الأمان.
  - قبول: عند الضغط يتم فتح `Security Setup`.

## 2️⃣ إعداد نظام الحماية

- خيارات: PIN (4/6)، كلمة مرور نصية، رسم نمط، بصمة، بصمة وجه.
- متطلبات:
  - UI لانتقاء الطريقة وإدخال/تأكيد السر.
  - تنفيذ محلي لـ validation و strength checks.
  - تخزين بيانات الاسترداد/ملامح البصمة لا يحوي كلمات السر نفسها؛ بدلاً من ذلك: اشتقاق مفتاح (Argon2id) ولف Master Key بواسطة Keystore.
- تحقق:
  - اختبار الوحدة: validation للـ PIN/Password/Pattern.
  - اختبار الأمان: Argon2 derivation يتم في native، لا تسرب كلمات المرور إلى سجلات.
  - Acceptance: يمكن للمستخدم إنشاء رمز ناجح وتأكيده.

## 3️⃣ إعداد طريقة استرداد الحساب (اختياري)

- خيارات: سؤال أمان أو رمز استرداد.
- متطلبات:
  - واجهة اختيار وتخزين آمن للمؤشر (لا تخزن الجواب كنص عادي بلا تشفير).
  - ربط خيار الاسترداد بتدفق استرجاع الحساب.
- تحقق:
  - حفظ سؤال الأمان كـ hashed (مع salt) داخل DB المشفّر.
  - اختبار استعادة الحساب عبر السؤال أو رمز الاسترداد.

## 4️⃣ شاشة الصلاحيات

- الأذونات المطلوبة: الوصول للصور/فيديو/ملفات (SAF)، استخدام البصمة، عرض فوق التطبيقات (اختياري)، إلخ.
- متطلبات:
  - طلب الأذونات عند الحاجة فقط، مع نص توضيحي (privacy-first).
  - التزام بـ Scoped Storage وSAF على Android 10–14.
- تحقق:
  - اختبار يدوي على أجهزة Android 10/11/12/13 للتأكد من سلوك الطلبات وfallback عند الرفض.

## 5️⃣ الواجهة الرئيسية للخزنة

- أقسام: الصور، الفيديو، الملفات، الصوتيات، التطبيقات المخفية، المستندات، العناصر المقفلة.
- متطلبات:
  - قائمة مبوبة وتجزئة فعّالة (lazy loading وpagination).
  - الوصول للـ thumbnails مخزّنة مشفّرة.
- تحقق:
  - UI tests لعرض الأقسام والتنقل بينها.

## 6️⃣ إضافة ملفات إلى الخزنة (Import)

- تدفق: زر `إضافة` -> اختيار نوع -> استخدام SAF picker -> stream إلى native للتشفير chunked.
- متطلبات فنية:
  - استخدام SAF وstreams (لا تنسخ الملفات أولًا إلى المسار العام بلا تشفير).
  - تشفير متدفّق chunked (مثلاً 1 MiB) في native باستخدام session handle.
  - عند النجاح: ادراج ميتاداتا داخل SQLCipher.
- تحقق:
  - Integration test: استيراد ملف كبير (مثال 100MB) مع مراقبة الذاكرة والوقت.
  - Acceptance: الملفات المستوردة قابلة للعرض داخل الخزنة وفُقدت من معرض النظام عند اختيار حذف من المعرض.

## 7️⃣ عرض الملفات داخل الخزنة

- متطلبات:
  - دوال عرض تستخدم تدفقات فك التشفير native (no raw plaintext on disk).
  - مشاركة/استعادة يجب أن تطلب صلاحيات لازمة وتنسّق مع SAF.
- تحقق:
  - Unit/Integration: فتح ملف صورة وفك تشفيره على-stream وعرضه.

## 8️⃣ إخفاء التطبيقات

- متطلبات:
  - قائمة التطبيقات المثبتة ومقدرة على تعطيل launcher shortcut (يتطلب صلاحيات أو استخدام Device Admin / Accessibility حسب النهج).
  - حفظ حالة الإخفاء ضمن DB مشفّر.
- تحقق:
  - Manual test: إخفاء/إظهار تطبيق والتأكد من عدم ظهوره في launcher.

## 9️⃣ قفل التطبيقات (App Lock)

- متطلبات:
  - اندماج مع Accessibility أو usage stats أو native hooking حسب سياسة Android وسياسات Play Store.
- تحقق:
  - Manual test: محاولة فتح تطبيق مقفول تؤدي إلى شاشة قفل الخزنة.

## 🔟 الكاميرا السرية

- متطلبات:
  - واجهة كاميرا داخل التطبيق تحفظ مباشرة مشفّرة باستخدام same streaming pipeline.
  - لا حفظ مؤقت غير مشفّر على التخزين العام.
- تحقق:
  - Integration: التقاط صورة وتحقق من أنها تظهر داخل الخزنة فقط.

## 11️⃣ إخفاء التطبيق (Obfuscation)

- متطلبات:
  - بدائل واجهة (calculator, calendar) مع آلية secret code entry.
  - حماية من سهولة كشف الواجهة المزيفة (لا تعرض كلمات دالة في manifest أو activity labels الصريحة).
- تحقق:
  - Manual: تحويل الأيقونة واجراء فتح عبر كود سري يعمل.

## 12️⃣ إنذار عند محاولة اختراق

- متطلبات:
  - عند X محاولات فاشلة: التقاط صورة عبر كاميرا أمامية (التحقق من أذونات) + تسجيل الطابع الزمني + حفظ الحدث مشفّراً.
  - إرسال إشعار محلي للمستخدم (اختياري إرسال عن بعد يحتاج موافقة صريحة).
- تحقق:
  - Manual: إدخال رمز خاطئ والتحقق من توليد حدث وتخزين الصورة.

## 13–14️⃣ النسخ الاحتياطي / الاستعادة

- متطلبات:
  - دعم تصدير مشفّر (export) إلى Google Drive أو ملف مشفّر محليًا مع مفتاح استعادة محمي بالـ Keystore.
  - استعادة تقوم بإعادة تشفير الملفات في الهدف المحلي.
- تحقق:
  - Integration: عمل backup ثم restore إلى جهاز آخر والتأكد من صلاحية الملفات.

## 15️⃣ إعدادات الأمان

- متطلبات:
  - واجهة تغيير كلمة المرور تؤدي إلى إعادة لفّ Master Key وإدخال القيم الجديدة في SQLCipher.
  - تحسينات: تمكين/تعطيل البصمة، تغيير سؤال الأمان.
- تحقق:
  - Unit: تغيير كلمة المرور يؤدي إلى إمكانية فتح الخزنة بالمفتاح الجديد وفشل القديم.

## 16️⃣ تسجيل الخروج/قفل التطبيق

- متطلبات:
  - قفل تلقائي وإلغاء أي session native، وzeroize لكل المفاتيح في الذاكرة.
- تحقق:
  - Acceptance: بعد الخروج، إعادة فتح التطبيق تطلب إعادة المصادقة.

## متطلبات عامة للتوثيق/التطوير
- وثّق كل API native/JS في `API_CONTRACTS.md` مع رموز الأخطاء المحتملة.
- أضف acceptance-tests مُؤتمتة على CI: native `selfTest()`, DB encryption check, import/export roundtrip.
- اعتبارات الأداء: أضف benchmark صغيرة لـ Argon2 وتخزين النتائج في `PERFORMANCE_STRATEGY.md`.

## خطة تنفيذ مُقترحة (خطوات صغيرة متسلسلة)
1. تحديث الوثائق (هذا الملف) وربطها بالـ issue/tasks.
2. تنفيذ UI: `Splash`, `Onboarding`, `Security Setup` مع unit tests.
3. تنفيذ تدفق Permissions وSAF integration (import pipeline) مع اختبار استيراد ملف صغير.
4. تنفيذ وحماية native session/fetch (تمت جزئيًا) واختبار `selfTest()` على جهاز محلي.
5. تنفيذ باقي الميزات بالتدريج: hide apps, app-lock, secret camera.
6. CI: إضافة بناء native لمكتبات `libsodium`/`argon2`, وتشغيل اختبارات native.
